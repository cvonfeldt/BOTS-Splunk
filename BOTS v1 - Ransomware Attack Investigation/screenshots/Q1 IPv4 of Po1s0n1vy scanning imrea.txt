Q1: IPv4 of Po1s0n1vy scanning imreallynotbatman.com: Knew that any scan of the website would include the word "scan" in the alert/log, thought I might have to narrow it down further after but found only one source ip related, so we know it's 40.80.148.42. In the scans we see that the server responded with http status 200 to 2 vuln scans: one to the root directory (just loading the home page and running basic scans/parsing), and one of the IIS 8.3 shortname scan in directory. The rest of the scans had 400/404 codes meaning the server.rejected the request, but the 200 code means the server successfully responded with info to those two requests. (2photos)

Q2: Company that made the vulnerability scanner: We can see in #1 that the vuln scanner was made by acunetix: "ET SCAN Acunetix Accept HTTP Header detected scan in progress"

Q3: What CMS is imreallynotbatman.com using: I knew that any POST data to the site would include the CMS, so I simply queried index="botsv1" imreallynotbatman.com POST and we can clearly see that the CMS is joomla (2 photos)

Q4: Name of file that defaced the website: First I thought it would be related to a http POST (assumed it was a web injection) and also assumed it would have poisonivy somewhere in it since attackers love notoriety, so I queried index="botsv1" poisonivy POST, but got no results. I realized they could've actually defaced the website via joomla access, so I removed the POST and found the file: poisonivy-is-coming-for-you-batman.jpeg, along with the confirmation that it indeed server-side and not an http injection. When removing the sourcetype filter to see fortigate logs as well, we see the batman webserver here is reaching out to the jumpingcrab 23.22.63.114 web server for the defacement file. (2 photos)

Q5: FQDN associated with the dynamic DNS attack: We can see in #4 the website but just to confirm, I added DNS to the query. (1 photo)

Q6: IPv4 Po1s0n1vy tied to pre-staged domains: This is just asking which server had the defacement attack ready to send to the batman webserver, which we already know from #4 is 23.22.63.114. 

Q7: IPv4 attempting brute force against imreallynotbatman.com: Know that any login attempt will send a POST and that the form_data will include "passwd" (and that a brute force will have a high volume of them), so used this query to find count of src_ip's sending POSTs to imreallynotbatman, where it was obviously 23.22.63.114 - all to the joomla admin. Also when looking at the one login attempt from 40.80.148.42 on the joomla admin, we can see that it was successful and lasting as it's connection_type is "keep_alive" where all of the brute force attempts are "closed." I filtered the query (adding correct password (batman) to passwd form data) to track down where the brute force finally tried the correct combo to see how it differed from the other brute force attempts, and there was no difference. All of the attempts (including the successful one from 40.80.148.42) had the http code of 303, meaning they were redirected to another site where I'm sure they were either authorized or denied. (2 photos)

Q8: What is the name of the executable uploaded by Po1s0n1vy?: First I assumed this would've been uploaded the same way the image defaced imreallynotbatman.com - server side from being sent from attacker web server to batman server, so I queried index="botsv1" src_ip="192.168.250.70" .exe dest_ip="23.22.63.114", but didn't find anything suspicious, so I then decuded it must've been http rather than server-side, so I queried: index="botsv1" .exe POST src_ip="40.80.148.42" dest_ip="192.168.250.70" (using the reasoning I was initially trying to use in #4 - that to change the site it would require a POST from the attacker machine to batman webserver). In this we can confirm with the upload of the .exe with "http_refer: http://imreallynotbatman.com/joomla/administrator/index.php?option=com_extplorer&tmpl=component" and "url: /joomla/administrator/index.php," that the attacker did indeed gain access to the admin controls of joomla. I'm thinking they uploaded the .exe to gain persistence/a backdoor and to automate the tasks they wanted for the attack, including the GET request to the attacker server for the defacement file. (2 photos)

Q9: MD5 hash of the uploaded executable: So to find this, I knew that I would need to search for the md5 where 3791 was the process, so I simply queried 'index="botsv1" process="3791.exe" source="WinEventLog:Microsoft-Windows-Sysmon/Operational"', but then found that obviously included processes like image loads and process terminations which all have their own unique MD5s. I knew I needed the original process creation (EventCode=1) where the child is 3791.exe and the parent is the cmd.exe. and found the md5 of AAE3F5A29935E6ABCC2C2754D12A9AF0. (1 photo)

Q10: SHA256 hash of the spear phishing malware: Had to go outside of splunk to get this one since this malware was on the atacker's server and was never transmitted over the network. Went to threatminer and found an md5 associated with the 23.22.63.114 attacking server, which when clicking it (hyperlink) led to me here where I found the SHA256. (1 photo)

Q11: Special hex code associated with customized malware: Entering the SHA256 we got from #10 into virustotal.com, one of the comments so graciously commented the hex code! (1 photo)

Q12: First brute force password used: Used the query that we used from #7 to find all brute force attempts, then sorted them by time to find the event of the first attempt, then found 12345678 in the formdata. (1 photo)

Q13: Six character Coldplay song used as password: I'm actually somewhat of a coldplay fan myself, and I can think of a few of their popular songs that are 6 letters off the top of my head, including "clocks", "sparks", "yellow", and "fix you," but "fix you" would be 7 characters with the space. Knowing these songs, I used this query to find all 6 letter passwords in the bruteforce, and if any of them were clocks, yellow, or sparks, they would appear at the top of the table. Q14: Correct admin password for the CMS: We know from #7 that this is "batman", but if I were to fully extract it with a query I'd use this (below) since we know that 40.80.148.42 made only one login attempt which was successful. (1 photo)

Q15: Average password length used in brute force: First I was trying this query with jsut the raw log data for the field type in the rex and was wondering why I was getting such weird lengths (like 20-40 characters) and in turn a skewed average length. I dind't think much of the field type at first because it had worked for me so far in previous answers in extracting passwords and hadn't noticed anything weird. Anyways, I printed out the passwords I was getting in a table and saw all of the extra characters the rex was taking from non form_data fields where the password entered in the POST didn't cease after & or whitespace. Once I specified form_data I got my average of baout 6.175 chars per password in the bruteforce. (2 photo)

Q16: Seconds elapsed between correct password found and compromised login: Went simple here and just found the moment of compromised login with the only attempt from 40.80.148.42 (9:48:05.858 PM), then took the time that the web server found the batman password via brute force (9:46:33.689 PM), and simply subtracted to get 92.17 seconds. (2 photos)
